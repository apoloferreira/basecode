SageMaker · Batch Transform

Tipos de Container

O SageMaker Batch Transform suporta três estratégias de container para servir modelos em processamento em lote. Cada uma representa um ponto diferente no tradeoff entre conveniência, controle e complexidade operacional.

DLC Puro

Deep Learning Container

Container gerenciado pela AWS. Zero Dockerfile. Zero código de serving. O modelo é servido pelo framework nativo do DLC (TorchServe, TF Serving…).

Dockerfile✗ Não
ECR push✗ Não
Código custom✗ Não
Cold startRápido
FlexibilidadeBaixa
DLC + Script Mode

Script Mode

Mesma imagem gerenciada, mas com código de inferência customizado empacotado no model.tar.gz. Contrato de 4 funções Python.

Dockerfile✗ Não
ECR push✗ Não
Código custom✓ Sim
Cold startMédio (pip install)
FlexibilidadeMédia
BYOC

Bring Your Own Container

Dockerfile customizado, servidor HTTP próprio (/ping + /invocations). Controle total sobre o runtime, dependências e formato de I/O.

Dockerfile✓ Sim
ECR push✓ Sim
Código custom✓ Sim
Cold startRápido (baked)
FlexibilidadeTotal

Quando usar cada abordagem

DLC Puro — escolha quando
  • O modelo é exportado em um formato nativamente suportado pelo DLC (TorchScript, SavedModel, ONNX via TF…)
  • Não há pré-processamento ou pós-processamento customizados além do que o framework oferece
  • A equipe não tem experiência em Docker e deseja o mínimo de atrito operacional
  • Prototipagem inicial onde a prioridade é velocidade de setup
DLC + Script Mode — escolha quando
  • O modelo precisa de lógica customizada de carregamento, pré ou pós-processamento
  • As dependências extras são leves (instaladas via pip em segundos) e o cold start não é crítico
  • Você quer evitar manter um Dockerfile e ciclo de build/push para o ECR
  • O formato de I/O (JSONL, campos específicos) não é suportado pelo DLC nativo
  • A lógica de inferência inclui acesso a recursos externos por record (ex: download de imagens do S3)
BYOC — escolha quando
  • As dependências são pesadas ou incompatíveis com o pip install em runtime (ex: CUDA toolkit específico, bibliotecas C++)
  • O cold start precisa ser o mais rápido possível (todas as deps estão baked na imagem)
  • O servidor HTTP precisa de endpoints extras além de /ping e /invocations
  • Você está usando um servidor de inferência especializado (Triton, TorchServe customizado, BentoML)
  • Há requisitos de compliance que exigem que a imagem seja auditada e imutável
  • O modelo está em formato não suportado pelo DLC (ex: ExportedProgram .pt2, ONNX via Triton ensemble)

DLC Puro Container gerenciado pela AWS — sem Dockerfile, sem ECR push. O modelo é servido diretamente pelo framework nativo (TorchServe, TF Serving…).
DLC Puro

Conceito

No modo DLC puro, você não escreve nenhum código de serving nem cria Dockerfile. A AWS fornece imagens pré-construídas para PyTorch, TensorFlow, MXNet e outros frameworks, com um servidor de inferência embutido que já expõe os endpoints /ping e /invocations que o SageMaker Batch Transform espera.

Para PyTorch, o DLC usa o TorchServe internamente. Para TensorFlow, o TF Serving. Você apenas entrega os artefatos do modelo em um model.tar.gz com a estrutura que o DLC espera, e o SageMaker cuida do resto.

Vantagens

  • Zero Dockerfile, zero ECR push
  • Manutenção mínima — basta atualizar o model.tar.gz
  • Cold start rápido (sem pip install em runtime)
  • A AWS atualiza e corrige a imagem
  • Ideal para protótipos rápidos

Limitações

  • Formato do modelo limitado ao que o DLC suporta
  • I/O fixo — o formato de entrada/saída é ditado pelo DLC
  • Sem lógica de pré/pós-processamento customizada
  • Sem acesso a recursos externos por record (ex: S3)
  • Difícil de debugar localmente sem replicar o TorchServe

Ciclo de vida do container

1 — Inicialização
Pull da DLC · Extração do model.tar.gz
O SageMaker faz pull da imagem DLC da AWS. O model.tar.gz é extraído em /opt/ml/model/. O TorchServe (ou equivalente) carrega o modelo.
↓
2 — Health check
GET /ping → 200
O DLC responde ao health check assim que o servidor de inferência está pronto. Nenhum código seu é executado.
↓
3 — Inferência
POST /invocations
O Batch Transform envia os dados no formato esperado pelo DLC. O TorchServe executa a inferência com o modelo carregado e devolve o resultado. Nenhuma função customizada é chamada.
↓
4 — Finalização
Container destruído
Ao fim do job, o container é destruído. Os artefatos do modelo em S3 permanecem intactos.

Estrutura do model.tar.gz

Para o DLC PyTorch com TorchServe, o model.tar.gz pode conter um .mar (Model Archive) gerado pelo torch-model-archiver, ou diretamente um arquivo .pt (TorchScript) com um handler padrão.

model.tar.gz ├── model.pt ← TorchScript compilado (torch.jit.script/trace) └── config/ └── model_config.yaml ← configuração do TorchServe (opcional)
⚠️
A estrutura exata do model.tar.gz varia por DLC e versão. Consulte a documentação da AWS para o framework e versão específicos que você está usando. O DLC PyTorch via TorchServe tem requisitos diferentes do DLC TensorFlow via TF Serving.

DLC + Script Mode

Conceito

O Script Mode usa a mesma imagem DLC gerenciada, mas permite que você injete código Python de inferência customizado diretamente dentro do model.tar.gz. O DLC detecta a pasta code/ e a variável de ambiente SAGEMAKER_PROGRAM, e chama suas funções Python no lugar do handler padrão.

Isso elimina a necessidade de Dockerfile e ECR push, mas dá controle completo sobre como o modelo é carregado, como os dados de entrada são desserializados e como os resultados são serialados de volta ao SageMaker.

Exemplo neste projeto: app/pytorch/inference.py — serve o modelo ResNet18 carregado via state_dict (.pth), com download de imagens do S3 por record e saída em JSONL.

Vantagens

  • Sem Dockerfile nem ECR push
  • Lógica de pré/pós-processamento completamente customizada
  • Dependências extras via requirements.txt no tar.gz
  • Qualquer formato de modelo suportado pelo Python
  • Testável localmente sem Docker
  • A AWS gerencia patches de segurança da imagem base

Limitações

  • pip install em runtime: cold start mais lento
  • Dependências pesadas (ex: CUDA-specific libs) podem ser problemáticas
  • Limitado às 4 funções do contrato — sem endpoints extras
  • Sem controle sobre o servidor HTTP interno (TorchServe/SageMaker MMS)
  • Debugging requer simular o ambiente DLC localmente

Contrato das 4 funções

O DLC espera que inference.py exporte exatamente essas quatro funções. O pipeline é composto automaticamente:

Chamada UMA VEZ — ao subir o container
def model_fn(model_dir: str)
Carrega o modelo de /opt/ml/model/ e retorna um objeto de contexto. Esse objeto é repassado como segundo argumento de predict_fn em todas as chamadas subsequentes.
↓
Cada POST /invocations — passo 1/3
def input_fn(request_body, content_type)
Desserializa o body HTTP recebido. Com SplitType=Line + SingleRecord, recebe uma linha por chamada. Com MultiRecord, pode receber N linhas. Deve retornar os dados no formato que predict_fn espera.
↓
Cada POST /invocations — passo 2/3
def predict_fn(data, model)
Executa a inferência. Recebe o output de input_fn e o contexto retornado por model_fn. É aqui que acontece download de S3, pré-processamento, inferência e pós-processamento.
↓
Cada POST /invocations — passo 3/3
def output_fn(prediction, accept)
Serializa o resultado para o body HTTP de resposta. Deve retornar uma tupla (body, content_type). O content_type vira o header Content-Type da resposta.
Python app/pytorch/inference.py — esqueleto comentado 
def model_fn(model_dir: str) -> dict:
    """Chamada 1x ao subir. model_dir = /opt/ml/model/"""
    model = ResNet18Model()
    model.load_state_dict(torch.load(f"{model_dir}/ResNet18.pth", weights_only=True))
    model.eval()
    return {"model": model, "device": device}


def input_fn(request_body: bytes, content_type: str) -> list[dict]:
    """Desserializa o JSONL recebido. Retorna lista de dicts."""
    lines = request_body.decode().strip().splitlines()
    return [json.loads(ln) for ln in lines if ln.strip()]


def predict_fn(data: list[dict], model_ctx: dict) -> list[dict]:
    """Download S3 + inferência para cada record."""
    results = []
    for record in data:
        img    = s3_utils.download_image(record["image_s3_path"])
        tensor = transform(img).unsqueeze(0).to(model_ctx["device"])
        with torch.no_grad():
            scores = model_ctx["model"](tensor)[0]
        results.append({"record_id": record["record_id"], "class": CLASSES[scores.argmax()]})
    return results


def output_fn(prediction: list[dict], accept: str) -> tuple[str, str]:
    """Serializa como JSONL. Retorna (body, content_type)."""
    body = "\n".join(json.dumps(r) for r in prediction)
    return body, "application/jsonlines"

Ciclo de vida do container

1 — Pull e extração
DLC pull · tar.gz extraído em /opt/ml/model/
O SageMaker faz pull da DLC (ex: pytorch-inference:2.5.1-gpu-py311). O model.tar.gz é extraído: artefatos do modelo na raiz e scripts em code/.
↓
2 — Instalação de dependências
pip install -r /opt/ml/model/code/requirements.txt
O DLC detecta requirements.txt em code/ e instala as dependências. Isso ocorre a cada cold start e pode levar de segundos a minutos dependendo dos pacotes.
↓
3 — Carregamento do modelo
model_fn("/opt/ml/model") chamada
O DLC importa inference.py (definido por SAGEMAKER_PROGRAM) e chama model_fn. O retorno é armazenado em memória e repassado para predict_fn em todas as chamadas.
↓
4 — Inferência (loop)
input_fn → predict_fn → output_fn
O Batch Transform envia N requests POST /invocations (MaxConcurrentTransforms em paralelo). Para cada request: input_fn desserializa → predict_fn executa → output_fn serializa a resposta.
↓
5 — Finalização
Container destruído · deps descartadas
Ao fim do job, o container é destruído. As dependências instaladas via pip são descartadas junto — serão reinstaladas no próximo cold start.
⚠️
Cold start e pip install: o pip install ocorre uma única vez por inicialização do container, não por record processado. O custo é pago no começo do job. Para dependências leves (boto3, Pillow), o overhead é de segundos. Para pacotes grandes (onnxruntime, scipy, etc.), pode ser de minutos — nesse caso, considere BYOC.

Opções de configuração

Variáveis de ambiente — Script Mode

VariávelDefaultObrigatórioDescrição
SAGEMAKER_PROGRAM inference.py SIM Nome do arquivo Python dentro de code/ que contém as 4 funções. Se não definida, o DLC usa inference.py por convenção.
SAGEMAKER_SUBMIT_DIRECTORY /opt/ml/model/code NÃO Diretório onde o DLC procura o script. Normalmente não é necessário alterar.
SAGEMAKER_MODEL_SERVER_WORKERS 1 NÃO Workers paralelos do servidor interno (Multi-Model Server). Aumentar pode conflitar com MaxConcurrentTransforms.
SAGEMAKER_MODEL_SERVER_TIMEOUT 60 NÃO Timeout (segundos) para cada chamada ao container. Aumente se predict_fn fizer downloads lentos.

create_transform_job — configurações recomendadas para Script Mode

ParâmetroValor típicoObservação
BatchStrategy SingleRecord Mais simples: cada linha JSONL vira uma chamada ao container. predict_fn recebe uma lista com 1 item. Use MultiRecord se quiser batching manual em predict_fn.
MaxConcurrentTransforms 4–8 Requests paralelos ao container. Como o servidor interno é síncrono por default, valores altos podem não ajudar sem aumentar SAGEMAKER_MODEL_SERVER_WORKERS.
MaxPayloadInMB 1–6 Com SingleRecord e linhas JSONL pequenas, 1 MB é suficiente.
SplitType Line O SageMaker divide o arquivo JSONL por linha antes de enviar ao container.
AssembleWith Line O SageMaker concatena as respostas linha a linha no arquivo de saída S3.
ContentType application/json Passado como header Content-Type. Verifique o que sua input_fn espera.

BYOC

Bring Your Own Container

No modo BYOC você constrói e mantém sua própria imagem Docker, publicada no Amazon ECR. O SageMaker não impõe nenhum framework ou servidor — a única exigência é que o container exponha os endpoints HTTP corretos na porta 8080.

Isso permite usar qualquer stack de serving: FastAPI, Flask, BentoML, NVIDIA Triton, TorchServe customizado, ou até um servidor escrito do zero. As dependências são instaladas na imagem (build time), eliminando o overhead de pip install a cada cold start.

Exemplos neste projeto:

  • app/pt2/ — FastAPI + uvicorn servindo um modelo ResNet18 em formato ExportedProgram (.pt2)
  • app/onnx/ — NVIDIA Triton com ensemble pipeline (preprocess → classifier → postprocess) + proxy Python para adaptar o protocolo KServe v2 ao SageMaker
  • app/triton_aws/ — DLC AWS sagemaker-tritonserver como imagem base, com boto3 e Pillow instalados via Dockerfile

Vantagens

  • Controle total sobre o servidor HTTP e o runtime
  • Dependências baked: cold start rápido
  • Formatos de modelo ilimitados
  • Endpoints extras (/health, /metrics, /execution-parameters)
  • Permite servidores especializados (Triton, TorchServe)
  • Imagem auditável e imutável (compliance)

Limitações

  • Dockerfile, build e push para ECR obrigatórios
  • Rebuild a cada mudança de dependência
  • Você é responsável pelos patches de segurança da imagem
  • Você deve implementar /ping e /invocations corretamente
  • Ciclo de desenvolvimento mais longo

Contrato HTTP

O SageMaker se comunica com o container via HTTP na porta 8080. Os dois endpoints obrigatórios são:

EndpointMétodoObrigatórioComportamento esperado
/ping GET SIM Retorna HTTP 200 quando o modelo está carregado e pronto para inferência. Retorna 503 enquanto inicializa. O SageMaker usa isso como health check antes de enviar requisições de inferência.
/invocations POST SIM Recebe o payload do Batch Transform e retorna os resultados. O corpo e o Content-Type são controlados pelas configurações do job (ContentType, SplitType, BatchStrategy).
/execution-parameters GET OPCIONAL O container pode sugerir MaxConcurrentTransforms, BatchStrategy e MaxPayloadInMB via JSON. Se implementado, o SageMaker usa esses valores como defaults do job.
Python app/pt2/main.py — FastAPI implementando o contrato 
from fastapi import FastAPI, Request, Response
from contextlib import asynccontextmanager

@asynccontextmanager
async def lifespan(app):
    app.state.predictor = Predictor()
    app.state.predictor.load()   # carrega o modelo UMA VEZ
    yield

app = FastAPI(lifespan=lifespan)

@app.get("/ping")
def ping(request: Request):
    # 200 = pronto, 503 = não pronto
    if not request.app.state.predictor.is_ready():
        return Response(status_code=503)
    return Response(status_code=200)

@app.get("/execution-parameters")
def execution_parameters():
    # Sugere configurações ao SageMaker
    return {
        "MaxConcurrentTransforms": 1,
        "BatchStrategy":           "MULTI_RECORD",
        "MaxPayloadInMB":          6,
    }

@app.post("/invocations")
async def invocations(request: Request):
    body  = await request.body()
    lines = [ln for ln in body.decode().strip().split("\n") if ln.strip()]
    results = [predictor.predict(json.loads(ln)) for ln in lines]
    output  = "\n".join(json.dumps(r) for r in results)
    return Response(content=output, media_type="application/jsonlines")
💡
Triton como BYOC: quando o servidor de inferência é o NVIDIA Triton, o protocolo nativo é o KServe v2, não o SageMaker HTTP. O padrão neste projeto (app/onnx/) é adicionar um proxy Python leve que converte: GET /ping → GET /v2/health/ready e POST /invocations → POST /v2/models/pipeline/infer. A DLC da AWS (sagemaker-tritonserver) já inclui esse proxy embutido, eliminando a necessidade de implementá-lo manualmente.

Ciclo de vida do container

1 — Pull e start
docker pull ECR · docker run image serve
O SageMaker faz pull da imagem do ECR e executa o container com o argumento serve (via CMD ou ENTRYPOINT no Dockerfile). O model.tar.gz é extraído em /opt/ml/model/.
↓
2 — Inicialização do servidor
Servidor HTTP sobe na porta 8080
O servidor customizado (FastAPI/uvicorn, Triton, Flask…) inicializa. O modelo é carregado de /opt/ml/model/. Todas as dependências já estão disponíveis (baked na imagem).
↓
3 — Health check
GET /ping → 200
O SageMaker aguarda /ping retornar 200 antes de iniciar os requests de inferência. Implemente 503 enquanto o modelo não estiver carregado para evitar requests prematuros.
↓
4 — Inferência (loop)
POST /invocations × MaxConcurrentTransforms
O Batch Transform envia requests paralelos. Seu servidor processa e responde. Com MultiRecord, cada request pode conter múltiplos registros. A resposta deve ter o mesmo número de linhas do input.
↓
5 — Finalização
Container destruído
Ao fim do job o container é destruído. Como as dependências estão baked na imagem, não há custo de reinstalação no próximo cold start.
