01Pré-requisitos

O Triton Python backend impõe três restrições que não são configuráveis:

model_name/
├── config.pbtxt # backend: "python" obrigatório
└── 1/
    └── model.py # nome fixo — o Triton não aceita outro
⚠
módulo disponível apenas no container

triton_python_backend_utils existe somente dentro do container Triton. Warnings de linter no ambiente local são falsos positivos esperados.

O config.pbtxt mínimo para Python backend:

protobuf · config.pbtxt
name: "meu_modelo"
backend: "python"
max_batch_size: 8

input [{ name: "entrada"  data_type: TYPE_FP32  dims: [3, 256, 256] }]
output[{ name: "saida"    data_type: TYPE_FP32  dims: [6]          }]

02O Contrato: TritonPythonModel

A classe deve ter exatamente o nome TritonPythonModel. O Triton instancia ela por reflection — qualquer outro nome silencia o backend sem mensagem de erro.

1× — container sobe single-threaded
def initialize(self, args: dict):
Ponto de inicialização. Abra conexões de rede, carregue arquivos auxiliares e instancie objetos reutilizados por execute(). É single-threaded — sem risco de condição de corrida.
O que fazer aqui
  • Abrir clientes de rede (boto3, httpx, grpc)
  • Carregar labels, configs e arquivos auxiliares
  • Ler variáveis de ambiente e validar valores obrigatórios
  • Instanciar objetos thread-safe para compartilhar com execute()
  • Logar aviso se configuração estiver incompleta
Chaves do dict args
args["model_config"] str (JSON) config.pbtxt serializado — parse com json.loads()
args["model_name"] str nome do modelo no repositório Triton
args["model_repository"] str path absoluto do model_repository no container
args["model_version"] str versão carregada, ex: "1"
N× por requisição thread-safe obrigatório
def execute(self, requests: list) → list[InferenceResponse]:
Núcleo do backend. Recebe lista de InferenceRequest e deve retornar lista de InferenceResponse de exatamente o mesmo tamanho. Pode ser chamado por múltiplas threads — objetos de initialize() devem ser thread-safe.
Regras do contrato
  • len(responses) == len(requests) — sempre, inclusive em erros
  • Nunca levante exceção no corpo — use TritonError()
  • Isole erros por request — um erro não derruba o lote
  • Shapes numpy devem bater com dims do config
  • dtype numpy deve corresponder ao data_type declarado
API pb_utils disponível em execute()
get_input_tensor_by_name(req, name)→ Tensorobtém tensor de entrada
tensor.as_numpy()→ ndarrayshape (batch, *dims)
Tensor(name, array)→ Tensorcria tensor de saída
InferenceResponse(output_tensors=[…])→ Responseresposta de sucesso
TritonError(msg)→ Errorerro isolado por request
InferenceResponse([], err=error)→ Responseresposta de erro
1× — container desce opcional
def finalize(self):
Ponto de limpeza. Feche tudo que foi aberto em initialize(). Recomendado sempre que recursos foram alocados.
O que fazer aqui
  • Fechar clientes HTTP, gRPC ou de banco de dados
  • Desalocar modelos carregados em memória
  • Encerrar threads ou processos auxiliares
  • Liberar handles de arquivos abertos
Assinatura
self — sem argumentos além de self
retorno None valor ignorado pelo Triton

Esqueleto completo

Python · model.py
import numpy as np
import triton_python_backend_utils as pb_utils


class TritonPythonModel:

    def initialize(self, args: dict):
        # args["model_config"]     → JSON str do config.pbtxt
        # args["model_name"]       → nome do modelo
        # args["model_repository"] → path no container
        # args["model_version"]    → versão carregada (ex: "1")
        pass

    def execute(self, requests: list) -> list:
        responses = []
        for request in requests:
            try:
                inp    = pb_utils.get_input_tensor_by_name(request, "nome_input")
                arr    = inp.as_numpy()        # ndarray (batch, *dims)
                result = arr * 2.0
                out    = pb_utils.Tensor("nome_output", result)
                responses.append(pb_utils.InferenceResponse(output_tensors=[out]))
            except Exception:
                err = pb_utils.TritonError("erro interno")
                responses.append(pb_utils.InferenceResponse(output_tensors=[], err=err))
        return responses  # DEVE ter o mesmo len que requests

    def finalize(self):
        pass

03Regras Críticas

críticolen(responses) == len(requests)O Triton trava silenciosamente se os tamanhos diferirem. Retorne sempre um response por request — inclusive em erro.
críticoNome da classe: TritonPythonModelCarregado por reflection. Qualquer variação silencia o backend sem mensagem de erro.
críticoNome do arquivo: model.pyO Triton procura exatamente este nome no diretório de versão (1/model.py). Não é configurável.

atençãodtype deve bater com data_typeTYPE_FP32 → np.float32 · TYPE_INT64 → np.int64 · TYPE_BYTES → np.object_

04TYPE_BYTES — Atenção Especial

O tipo TYPE_BYTES é representado como np.object_. O comportamento difere de todos os tipos numéricos.

Leitura — input BYTES

Python · lendo TYPE_BYTES
# config.pbtxt: dims: [1]  →  shape numpy: (batch_size, 1), dtype object
raw = pb_utils.get_input_tensor_by_name(request, "s3_uri").as_numpy()
for i in range(len(raw)):
    uri: str = raw[i, 0].decode("utf-8")   # bytes → str

Escrita — output BYTES

Python · escrevendo TYPE_BYTES
# dtype DEVE ser np.object_ — nunca np.bytes_ ou str
class_names = np.array(
    [[b"cardboard"], [b"glass"]],   # lista de bytes Python
    dtype=object                         # shape: (batch_size, 1)
)
pb_utils.Tensor("class_name", class_names)
ℹ
max_batch_size e a dimensão de batch

Com max_batch_size > 0, o Triton adiciona a dimensão de batch automaticamente. dims: [1] no config → shape numpy (batch_size, 1).

05Dependências Python Extras

O Triton não instala pacotes por modelo. Qualquer import além da stdlib e triton_python_backend_utils deve estar na imagem Docker:

Dockerfile
FROM nvcr.io/nvidia/tritonserver:24.12-py3
RUN pip install --no-cache-dir \
    "boto3>=1.34"   \
    "Pillow>=10.0"
    # "pacote>=versão"
⚠
Python backends × backend onnxruntime

Python backends precisam dos pacotes instalados via pip. O backend onnxruntime usa a biblioteca C++ nativa do Triton — não precisa de pip.

config.pbtxt
Referência Completa

Todo modelo no Triton precisa de um config.pbtxt no formato Protocol Buffers. Este arquivo declara o backend, os tensores de entrada e saída, o tamanho do batch e o comportamento de scheduling.

01Campos Raiz do Modelo

name string opcional
Nome do modelo. Deve coincidir com o nome do diretório no model_repository. Se omitido, o Triton usa o nome do diretório.
Boa prática: sempre declarar explicitamente para evitar dependência da estrutura de diretórios.
Exemplo
name: "preprocess"
backend string condicional
Especifica qual backend carrega e executa o modelo. Obrigatório para todos os modelos exceto ensemble (que usa platform em vez de backend).
"python"Python backend "onnxruntime"ONNX nativo "tensorflow" "pytorch" "tensorrt"
Exemplo
backend: "python"
# ou
backend: "onnxruntime"
platform string ensemble only
Usado exclusivamente para modelos ensemble. Substitui backend neste caso — os dois não coexistem no mesmo config.
"ensemble"pipeline de modelos
Exemplo
platform: "ensemble"
max_batch_size int obrigatório
Controla se o Triton adiciona uma dimensão de batch aos tensores e qual o limite máximo.
  • 0 — batching desabilitado. Os dims declarados são o shape completo do tensor.
  • > 0 — batching habilitado. O Triton adiciona a dimensão de batch automaticamente como primeira dimensão. Os dims declaram apenas as dimensões restantes.
⚠

Todos os modelos de um ensemble devem ter o mesmo max_batch_size.

Efeito no shape do tensor
max_batch_size: 0 dims: [3,256,256] → shape real é (3,256,256)
max_batch_size: 8 dims: [3,256,256] → shape real é (batch,3,256,256)

02Campos input [ ] e output [ ]

Cada tensor (entrada ou saída) é um bloco com três campos obrigatórios. O modelo pode ter múltiplos tensores de entrada e saída.

input [ ] / output [ ] repeated TensorSpec obrigatório
Campos de cada tensor
name string Nome do tensor. Deve ser idêntico ao nome usado no modelo (ONNX export, model.py, ensemble map).
data_type TYPE_* Tipo de dado. Ver tabela completa na seção 03.
dims int[] Shape excluindo a dimensão de batch (quando max_batch_size > 0). Use -1 para dimensão dinâmica.
⚠

O name do tensor ONNX deve coincidir exatamente com os input_names/output_names usados em torch.onnx.export().

Exemplos de dims
dims: [3, 256, 256] imagem RGB 256×256, shape fixo
dims: [6] vetor de scores com 6 classes
dims: [1] 1 string (TYPE_BYTES) por item
dims: [-1] comprimento dinâmico
dims: [-1, 512] sequência dinâmica × embedding 512

03Tipos de Dados (data_type)

Cada data_type no config.pbtxt corresponde a um dtype numpy específico no Python backend. A correspondência deve ser exata.

data_type (config) numpy dtype Bits Uso típico
TYPE_FP32 np.float32 32 Tensores de modelos ML — padrão para inferência
TYPE_FP16 np.float16 16 Inferência otimizada em GPU (TensorRT, mixed precision)
TYPE_FP64 np.float64 64 Alta precisão numérica — raro em produção
TYPE_INT32 np.int32 32 Índices, labels inteiras, token IDs
TYPE_INT64 np.int64 64 IDs longos, timestamps, contagens grandes
TYPE_INT16 np.int16 16 Valores inteiros compactos
TYPE_INT8 np.int8 8 Modelos quantizados INT8
TYPE_UINT8 np.uint8 8 Pixels de imagem (0–255) antes de normalizar
TYPE_BOOL np.bool_ 1 Máscaras de atenção, flags booleanas
TYPE_BYTES np.object_ variável Strings, URIs, JSON, payloads binários
⚠ dtype deve ser object, não bytes_ ou str
TYPE_STRING np.object_ variável Alias de TYPE_BYTES — comportamento idêntico

04instance_group [ ]

Define onde e quantas instâncias do modelo o Triton cria. Afeta diretamente onde o backend executa (CPU ou GPU) e o nível de paralelismo.

kind enum opcional
Valores disponíveis
KIND_AUTO GPU se disponível, senão CPU. Padrão quando omitido.
KIND_GPU Força GPU. Para ONNX backend, ativa CUDAExecutionProvider.
KIND_CPU Força CPU. Recomendado para Python backends com I/O (boto3, PIL).
KIND_MODEL Deixa o backend decidir o dispositivo (raro).
Exemplos
# Python backend que faz I/O — força CPU
instance_group [{ kind: KIND_CPU }]

# ONNX backend — auto-detecta GPU
# (omitir instance_group tem o mesmo efeito)
instance_group [{ kind: KIND_AUTO }]

# 2 instâncias na GPU 0 para alta throughput
instance_group [{
  kind: KIND_GPU
  count: 2
  gpus: [0]
}]

05dynamic_batching { }

Quando max_batch_size > 0, o Triton pode agregar requests individuais em batches para melhorar o throughput da GPU. O bloco dynamic_batching controla essa política.

dynamic_batching block opcional
Campos principais
preferred_batch_size int[] Tamanhos de batch preferidos. O Triton aguarda até atingir um desses tamanhos antes de enviar.
max_queue_delay_microseconds int Tempo máximo de espera (µs) para formar um batch. Após esse prazo, envia o batch mesmo incompleto.
Exemplo
dynamic_batching {
  preferred_batch_size: [4, 8]
  max_queue_delay_microseconds: 100
}
Com essa config: o Triton aguarda até juntar 4 ou 8 requests. Se em 100µs não atingir nenhum dos dois, despacha com o que tiver na fila.

06ensemble_scheduling { }

Exclusivo de modelos com platform: "ensemble". Define o grafo de execução: quais modelos rodam, em que ordem, e como os tensores fluem entre eles via input_map / output_map.

ensemble_scheduling block ensemble only
Campos de cada step
model_name string Nome do modelo a executar neste step (deve existir no model_repository).
model_version int -1 usa a versão mais recente disponível.
input_map map key: nome do tensor no modelo filho · value: nome do tensor no pipeline global.
output_map map key: nome do tensor de saída do modelo filho · value: nome no pipeline para uso nos próximos steps.
Como o roteamento funciona
O ensemble mantém um namespace global de tensores. Cada step lê tensores desse namespace via input_map e publica suas saídas de volta via output_map.
input pipeline: s3_uri
  → preprocess.input_map["s3_uri"] = "s3_uri"
  ← preprocess.output_map["preprocessed"] = "preprocessed"
  → classifier.input_map["input"] = "preprocessed"
  ← classifier.output_map["output"] = "scores"
  → postprocess.input_map["scores"] = "scores"
  ← postprocess.output_map["class_name"] = "class_name"
output pipeline: class_name, score

07Exemplos — Projeto Atual

Os quatro config.pbtxt que compõem o pipeline ResNet18 ONNX neste repositório.

preprocess — Python backend

preprocess/config.pbtxt
name: "preprocess"
backend: "python"
max_batch_size: 8

input [{
  name: "s3_uri"
  data_type: TYPE_BYTES    # URI S3 como string
  dims: [1]               # 1 URI por item do batch
}]

output [{
  name: "preprocessed"
  data_type: TYPE_FP32
  dims: [3, 256, 256]    # (C, H, W) sem dim de batch
}]

instance_group [{ kind: KIND_CPU }]   # I/O-bound, não precisa de GPU

classifier — ONNX backend

classifier/config.pbtxt
name: "classifier"
backend: "onnxruntime"
max_batch_size: 8

input [{
  name: "input"            # deve coincidir com input_names do torch.onnx.export
  data_type: TYPE_FP32
  dims: [3, 256, 256]
}]

output [{
  name: "output"           # deve coincidir com output_names do torch.onnx.export
  data_type: TYPE_FP32
  dims: [6]               # 6 classes: cardboard, glass, metal, paper, plastic, trash
}]

# instance_group omitido → KIND_AUTO (GPU se disponível)

postprocess — Python backend

postprocess/config.pbtxt
name: "postprocess"
backend: "python"
max_batch_size: 8

input [{
  name: "scores"
  data_type: TYPE_FP32
  dims: [6]
}]

output [
  {
    name: "class_name"
    data_type: TYPE_BYTES   # np.object_ com bytes no Python backend
    dims: [1]
  },
  {
    name: "score"
    data_type: TYPE_FP32
    dims: [1]

instance_group [{ kind: KIND_CPU }]

pipeline — Ensemble

pipeline/config.pbtxt
name: "pipeline"
platform: "ensemble"       # usa platform, não backend
max_batch_size: 8           # deve ser igual em todos os modelos do ensemble

input [{ name: "s3_uri"     data_type: TYPE_BYTES  dims: [1] }]

output [
  { name: "class_name"  data_type: TYPE_BYTES  dims: [1] },
  { name: "score"       data_type: TYPE_FP32   dims: [1] }
]

ensemble_scheduling {
  step [
    {
      model_name: "preprocess"
      model_version: -1
      input_map  { key: "s3_uri"      value: "s3_uri"       }
      output_map { key: "preprocessed" value: "preprocessed" }
    },
    {
      model_name: "classifier"
      model_version: -1
      input_map  { key: "input"  value: "preprocessed" }
      output_map { key: "output" value: "scores"       }
    },
    {
      model_name: "postprocess"
      model_version: -1
      input_map  { key: "scores"     value: "scores"     }
      output_map { key: "class_name" value: "class_name" }
      output_map { key: "score"      value: "score"      }
    }
  ]
}
